Step 2 — Create 2 MongoDB models

Model A: Telemetry
Fields:

device_id, shipment_id

ts

temp_c, humidity

lat, lng

tamper_lid_open, shock_flag

derived flags: cold_chain_violation, route_violation, tamper_violation

Model B: AuditLog (hash-chained)
Fields:

ts, event_type, payload_json

prev_hash

hash = SHA256(prev_hash + payload_json + ts)

This gives you “blockchain behavior” (tamper-evident history) without a blockchain dependency, which matches the repo reality (no chain libs).

Step 3 — Build a Synthetic Device Simulator Script (Python or Node)

This script will “act like” your ESP32 + sensors:

What it should simulate (realistic, judge-proof)

Cold chain temperature: stable 4–6°C, then an excursion spike (e.g., 12°C for 5 mins), then recover.

GPS route: a list of lat/lng points interpolated every 2 seconds.

Tamper events: lid_open for 20 seconds, or shock_flag once.

Data packet format (example)
{
  "device_id": "VCBOX-01",
  "shipment_id": "SHIP-2026-001",
  "ts": "2026-02-28T12:01:05Z",
  "temp_c": 5.3,
  "humidity": 61.2,
  "lat": 19.0760,
  "lng": 72.8777,
  "lid_open": false,
  "shock": 0
}

Your simulator just POSTs this to:

http://localhost:8000/iot/telemetry

Step 4 — Alert logic (keep it brutally simple)

Rules:

Cold chain violation if temp_c < 2 || temp_c > 8

Tamper if lid_open == true || shock == 1

Geofence violation if distance from route corridor > threshold (you can do a simplified version: “outside bounding box” for hackathon)

store telemetry
